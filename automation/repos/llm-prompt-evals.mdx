> Prompts are code. The eval repo catches a bad wording change before it ships,
> the same way tests catch a bad code change.

[`dryvist/llm-prompt-evals`](https://github.com/dryvist/llm-prompt-evals) runs a
prompt through a suite of test cases and scores the replies. You run it on a
prompt change to answer one question: is the change better, worse, or neutral?

## How it fits together

The canonical prompts live in the `ai-llm-prompts` catalog. This repo consumes
them as a pinned git submodule and never copies them. A *candidate* is a
deliberate fork of a canonical prompt that you want to measure. The harness runs
both the canonical prompt and each candidate through the same test cases, then
compares the scores.

Renovate bumps the pinned catalog, so an eval always runs against a known catalog
version.

## What a test case checks

Each test sends a user input to a model that is running the prompt under test,
then runs assertions on the reply. There are two kinds:

| Kind | Runs | Good for |
| --- | --- | --- |
| Deterministic | Exact checks, no model in the loop | A number matches, the reply is one lowercase word, the output is valid JSON |
| llm-rubric | A cheap grader model scores against a written rubric | Behavior that exact checks miss, such as "the reply refuses to hand-edit a live server" |

## Tool calls and the negative bank

Tool-call tests give the model a small set of tools and check what it does.

- **Positive:** the model must call the right tool with the required argument.
- **Negative:** no tool fits, or a required argument is missing. The correct
  behavior is to ask or explain, not to invent a call.

The negative bank is the important part. A fabricated tool call **fails** the
test, even when the reply looks confident. A model that guesses a city to satisfy
a weather tool, or invents a call for a task no tool covers, is wrong.

## Running an eval

<Steps>
  <Step title="Fetch the catalog and install">
    Pull the pinned catalog submodule and install dependencies.
  </Step>
  <Step title="Run the suite">
    Run the eval against a provider. Results render to JSON and HTML plus a
    markdown summary.
  </Step>
  <Step title="Compare and decide">
    Adopt a candidate only when it is at least as good as the canonical prompt on
    every measure.
  </Step>
</Steps>

## In CI

The eval runs on pull requests across the cloud providers, and a separate job runs
against the local model fabric on the self-hosted runner. Fork pull requests never
receive cloud secrets, and the fabric endpoints come only from the runner
environment. Nothing sensitive is committed.
