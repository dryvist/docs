# mintlify-mermaid-theme

Accept a high-level intent description ("show how X feeds Y feeds Z") and emit a complete, validated fenced Mermaid block using the Reef Green visual identity.

## When to use

- Adding a "How it fits" diagram to a docs page.
- Illustrating data flows, service relationships, or architectural dependencies.
- Any MDX page that calls for a visual — replace prose like "X sends data to Y which stores it in Z" with a Mermaid block.

## When NOT to use

- PNG/SVG rendering outside the Mintlify build pipeline.
- Replacing the upstream `obsidian-visual-skills:mermaid-visualizer` skill — this wraps it for docs-specific conventions.
- Producing diagrams outside a Mintlify MDX file context.

## Theme directive (pin on every diagram)

Every output must open with this `%%{init: ...}%%` directive, verbatim:

```
%%{init: {"theme": "base", "themeVariables": {"primaryColor": "#4FB3A9", "lineColor": "#E06B4A", "primaryTextColor": "#F4EFE6", "primaryBorderColor": "#2F7E78"}, "flowchart": {"look": "handDrawn", "defaultRenderer": "elk"}, "fontFamily": "Geist"}}%%
```

Do not omit or alter any key. If the diagram type is not `flowchart` (e.g., `sequenceDiagram`), drop the `flowchart` sub-key but retain all themeVariables.

## Shape vocabulary

| Concept | Mermaid syntax | When to use |
|---------|---------------|-------------|
| Service / process | `id(["Label"])` | Anything that runs code or transforms data |
| Data store | `id[("Label")]` | Databases, object stores, caches, lock files |
| Decision | `id{"Label"}` | Conditional branch or policy gate |
| Endpoint / source | `id(("Label"))` | External actors, CLI entry points, webhook origins |
| Logical grouping | `subgraph Title` ... `end` | Repo boundary, environment, team |

Deviate only when the concept genuinely does not map — and explain why.

## Style overrides

After the graph body, add one `style` rule per node:

- **Focal node** (the thing this diagram is about): `stroke:#E06B4A,stroke-width:4px`
- **Neighbor nodes**: `stroke:#2F7E78,stroke-width:2px`

Example:

```
style myService stroke:#E06B4A,stroke-width:4px
style neighborA  stroke:#2F7E78,stroke-width:2px
```

## Validation

After generating the block, pipe it through `mmdc` (available in the dev shell). Use a quoted heredoc — `echo` mangles single quotes and backticks that appear in real Mermaid blocks (e.g. `'primaryColor':'#4FB3A9'` and node labels):

```bash
nix develop --command mmdc --input - --output /dev/null <<'MERMAID'
<mermaid block>
MERMAID
```

If `mmdc` exits non-zero, fix the syntax error before returning output. If `mmdc`/`nix develop` is unavailable in the current environment, append `<!-- unvalidated -->` immediately after the fence — `mintlify-build-guard` will catch any errors at CI time. Never silently return a block that bypassed both layers of validation.

## Composing with other skills

- **`mintlify-page-author`** will call this skill to generate the `## How it fits` diagram on new pages. When called from that context, the focal node is the repo being documented.
- **`mintlify-build-guard`** validates every Mermaid block in the repo via `mmdc`. Blocks generated by this skill should pass that check automatically.

## Full workflow

1. **Receive intent.** The author describes the relationship in plain language.
2. **Identify nodes and edges.** Parse the intent into entities (nodes) and relationships (edges). Label edges concisely (e.g., `"loads"`, `"writes"`, `"feeds"`, `"deploys"`).
3. **Choose shapes.** Apply the shape vocabulary above.
4. **Emit diagram.** Write the full fenced block with the `%%{init}%%` directive and `style` overrides.
5. **Validate.** Run `mmdc` and fix any parse errors.
6. **Return.** Emit the final fenced block, ready to paste into an MDX file.

## Output format

Always return a fenced Mermaid block in this shape (the four-backtick outer fence below is just so the SKILL renders correctly — the actual output uses the standard three-backtick `mermaid` fence):

````
```mermaid
%%{init: ...}%%
flowchart LR
  ...
  style <focal> stroke:#E06B4A,stroke-width:4px
  style <neighbor> stroke:#2F7E78,stroke-width:2px
```
````

Use `LR` (left-to-right) for pipeline flows; `TD` (top-down) for hierarchical structures. Ask the author if orientation is ambiguous.
