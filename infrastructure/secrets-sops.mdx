> If it can be encrypted and committed, SOPS holds it. If it must rotate, Doppler does. If it must never touch disk, Keychain.

The homelab IaC stack runs a clean three-way split for secrets. This page covers the SOPS slice as it shows up in the Terraform and Ansible repos — what gets encrypted, who holds the age key, how it interacts with Doppler at runtime, and what the secrets policy actually enforces.

For the broader posture (every tool, every tier), see [Security overview](/security/overview) and the dedicated [SOPS tool page](/security/tools/sops). This page is the IaC-flavored cut.

## The three-way split

| Tier | Holds | Mechanism |
| --- | --- | --- |
| Encrypted-at-rest in git | Bootstrap creds that don't rotate, per-host variables, structured config | SOPS + age |
| Rotating runtime values | API tokens, HEC tokens, license keys, anything that gets reissued | Doppler |
| Local-only material | AI tool secrets, anything that must never reach a shell history | macOS Keychain |

The rule is mechanical: if a value rotates frequently, it belongs in Doppler — `git log` keeps encrypted history forever, and "encrypted today" is only as strong as the age key. If it must never leave the host, it belongs in Keychain. Everything else — Proxmox SSH keys, bootstrap passwords, per-env Terraform variables that name internal networks — goes in SOPS.

## How it shows up in Ansible repos

[`ansible-proxmox`](/infrastructure/ansible-proxmox), [`ansible-proxmox-apps`](/infrastructure/ansible-proxmox-apps), and [`ansible-splunk`](/observability/ansible-splunk) all follow the same pattern:

```yaml
# secrets.sops.yml.example — committed
NAS_HOMEASSISTANT_SMB_PASSWORD: change-me
```

```bash
cp secrets.sops.yml.example secrets.sops.yml
sops --encrypt --in-place secrets.sops.yml   # encrypts in place
git add secrets.sops.yml                     # safe: values are ciphertext now
```

At runtime, `sops exec-env secrets.sops.yml -- ansible-playbook ...` decrypts into a subprocess env only; nothing persists to disk. Rotating values stay in Doppler, accessed through `doppler run -- ansible-playbook ...`. The two wrappers compose cleanly — Doppler outermost, sops inside.

## How it shows up in Terraform repos

[`terraform-proxmox`](/infrastructure/terraform-proxmox) is the canonical example: a `.sops.json` under each environment folder, listed in `.sops.yaml` with the age recipient that controls it. Terragrunt consumes it through the `sops_decrypt_file` data source; resolved values land in Terraform state encrypted by the state backend's KMS key, not on disk.

`terraform-runs-on`, `terraform-aws-bedrock`, and `terraform-aws-static-website` use the same mechanism but keep the encrypted file smaller — those repos lean on Doppler for AWS creds and reserve SOPS for things like Bedrock model IDs and bootstrap config that doesn't change.

## The `.sops.yaml` configuration

Each repo declares which paths to encrypt with which keys at the repo root:

```yaml
creation_rules:
  - path_regex: \.sops\.json$
    age: >-
      age1aaaa... # public age recipient
```

Only the public half of the age key appears here. The private half lives at `~/.config/sops/age/keys.txt` (local convenience) and is escrowed in Bitwarden (canonical backup). The example shown is a placeholder — real public recipients live in each repo's `.sops.yaml`.

## What you must never commit

Per the secrets policy: no real IPv4 addresses, no real internal hostnames, no real domain names, no AWS account IDs, no SSH keys, no user-specific paths. Use placeholders or variables for anything tied to one user; every committed value should work for any person who clones the repo right now. The committed scrubbed-value table is `192.168.0.*`, `example.com`, `example.local`, `your-token-here`, generic role-based usernames.

## Rotating the age key

The five-step dance is the same across every repo that uses SOPS:

<Steps>
  <Step title="Generate the new key">
    `age-keygen -o ~/.config/sops/age/keys.txt.new`. Read-only file, not a service.
  </Step>
  <Step title="Add the new public recipient to every .sops.yaml">
    Keep the old recipient until cutover so neither party loses access mid-rotation.
  </Step>
  <Step title="Re-encrypt every SOPS file with both recipients">
    `sops updatekeys .sops.json` per file. The encrypted file changes; the plaintext doesn't.
  </Step>
  <Step title="Remove the old recipient">
    Re-run `sops updatekeys` once more. Now only the new key can decrypt.
  </Step>
  <Step title="Escrow + revoke">
    New key goes into Bitwarden the moment it's generated. Revoke the old key in any tooling that referenced it.
  </Step>
</Steps>
