> The normalization layer. Every dashboard reads through this TA's macros so the App layer never re-parses the raw JSON.

<RepoMeta language="Python" status="active" lastActive="this week" repoUrl="https://github.com/JacobPEvans/VisiCore_TA_AI_Observability" />

`VisiCore_TA_AI_Observability` is a Splunk Technology Add-on covering field extractions for 20 sourcetypes across Claude, Gemini, and Antigravity, with OTel GenAI semantic-convention aliases, Splunk CIM mappings, search macros, and a model-pricing CSV. It's invisible in the Splunk UI (`is_visible = false`) — only the [companion App](/observability/visicore-app) renders dashboards on top.

## Knowledge objects

| File | Holds |
| --- | --- |
| `props.conf` | Field extractions for 20 sourcetypes (Claude, Gemini, Antigravity) |
| `transforms.conf` | Lookup definitions |
| `macros.conf` | 9 reusable search macros (index filters, token extraction, cost calc, tools, dedup, cache) |
| `eventtypes.conf` | 5 event types for Claude and Gemini events |
| `tags.conf` | `ai`/`llm`/`genai` tags for CIM compliance |
| `lookups/` | Model pricing CSV for dollar-cost estimation |

## Token model

Aligned with [`ccusage`](https://github.com/ryoppippi/ccusage) — four token types:

| Token type | Field |
| --- | --- |
| Input | `input_tokens` |
| Output | `output_tokens` |
| Cache read | `cache_read_input_tokens` |
| Cache creation | `cache_creation_input_tokens` |

Dedup runs on the composite key `messageId:requestId`.

## OTel field mapping

Native fields are aliased to OpenTelemetry GenAI semantic-convention names so any OTel-aware downstream consumer (or a future migration off Splunk) gets a clean shape:

| Native | OTel |
| --- | --- |
| `message.model` | `gen_ai.response.model` |
| `sessionId` | `gen_ai.conversation.id` |
| `message.usage.input_tokens` | `gen_ai.usage.input_tokens` |
| `message.usage.output_tokens` | `gen_ai.usage.output_tokens` |
| `message.stop_reason` | `gen_ai.response.finish_reasons` |
| literal `"anthropic"` | `gen_ai.provider.name` |

## CIM mapping

A second mapping projects the same data into the Splunk Common Information Model so it interoperates with the Web and Change data models:

| Native | CIM |
| --- | --- |
| `type` | `action` |
| `message.model` | `dest` |
| `sessionId` | `session_id` |
| `"anthropic"` | `vendor` |
| `"claude_code"` | `product` |
| Tool name | `object`, `command` |
| `input.file_path` | `object_path` |

## Search macros

These are the contract the [App](/observability/visicore-app) builds on. Every dashboard runs through them:

| Macro | What it does |
| --- | --- |
| `` `claude_assistant_events` `` | Base filter for Claude assistant messages |
| `` `extract_tokens` `` | Extract the four token types as convenience fields |
| `` `calculate_cost` `` | Model-aware cost calculation against the pricing lookup |
| `` `extract_tools` `` | Tool-use extraction with CIM Change fields |
| `` `dedup_messages` `` | Dedup on `messageId:requestId` |
| `` `calculate_cache_pct` `` | Cache-hit percentage |

## How it fits

| Upstream | This | Downstream |
| --- | --- | --- |
| Cribl Stream HEC events at `index=claude` / `index=gemini` | Field extractions, aliases, macros, CIM tagging, lookups | [VisiCore App](/observability/visicore-app) dashboards; any user search |

## Getting started

<Steps>
  <Step title="Install the TA before the App">
    `splunk install app VisiCore_TA_AI_Observability-*.tar.gz && splunk restart`.
  </Step>
  <Step title="Verify indexes exist">
    `claude` and `gemini` must be provisioned. `ansible-splunk` owns that.
  </Step>
  <Step title="Sanity-check the extractions">
    `` `claude_assistant_events` | `extract_tokens` | head 5 `` should return events with token fields populated.
  </Step>
  <Step title="Install the companion App">
    Once the TA returns clean data, install [VisiCore App](/observability/visicore-app) for the dashboards.
  </Step>
</Steps>

## Packaging

`./scripts/package.sh` produces a versioned tarball in `build/`. CI validates with `jq . app.manifest`; `ansible-splunk` ships the tarball alongside the App.
